# Technical Architecture & Internals

Bento uses OS-level sandboxing primitives to enforce kernel-level isolation, network proxies for fine-grained network filtering, and strict filesystem controls. This document details how Bento operates internally.

---

## Technical Overview

Bento's sandboxing model applies constraints to the entire process tree of the executing script:

- **macOS**: Bento generates dynamic SBPL (Seatbelt) profiles passed directly to the native `sandbox-exec` utility.
- **Linux**: Bento uses `bubblewrap` for filesystem, PID, and mount namespacing. It couples this with kernel-level network filtering (Landlock TCP or network namespaces) and a custom `bento-launcher` shim to apply seccomp rules.

```
bento/
├── bento.go                    # public API (aliases + entry funcs)
├── doc.go                      # package docs
├── cmd/
│   ├── bento/                  # CLI entrypoint
│   └── bento-launcher/         # seccomp + Landlock shim binary (embedded)
└── internal/
    ├── spec/                   # manifest types + DangerousFiles
    ├── runner/                 # bwrap (Linux) + sandbox-exec (macOS)
    ├── proxy/                  # HTTP CONNECT + SOCKS5 + match logic
    ├── doctor/                 # environment health checks
    ├── sysprobe/               # find host binaries (bwrap, socat, ...)
    └── launcherbin/            # embedded launcher binary (regenerated by `make launcher`)
```

The `bento-launcher` is compiled as a separate `linux/amd64` binary and embedded into the Go package using `go:embed`. At runtime, the parent process extracts it to a temporary location, bind-mounts it inside the sandbox, and invokes it as the sandboxed environment's entrypoint.

The launcher installs seccomp and Landlock filters, then calls `execveat` to load the interpreter. Utilizing `execveat` ensures that the final state transition is permitted by the filters it has just loaded.

## Filesystem Isolation

Bento assumes a **deny-by-default** stance for the filesystem. A path must be explicitly allowed to be visible:

* **Linux (`bubblewrap`)**: Declared read-only paths are mounted via `--ro-bind`, and write paths via `--bind`. Always-blocked paths are masked by mounting `/dev/null` over them.
* **macOS (`sandbox-exec`)**: Bento compiles explicit `(allow file-read* (subpath ...))` and `(allow file-read* file-write* (subpath ...))` rules within an overall `(deny default)` Seatbelt profile.

### Symlink-Escape Rejection
To prevent race conditions and sandbox escapes, Bento traverses and `lstat`s every path component of all declared write directories before mounting. If any existing component is a symlink, the execution is immediately rejected. This mitigates attacks where a malicious workspace links `/tmp/out` to a sensitive host directory (like `~/.ssh`) in the split second between permission validation and mount initialization.

## Network Isolation

All outgoing network connections are intercepted and compared against the allowed list of domain/port rules defined in the manifest.

### Multi-Layer Filtering

| Layer | Targets | Mechanism |
|---|---|---|
| **HTTP CONNECT Proxy** | `curl`, Python `requests`, Node `http`, Go `net/http` | Intercepts HTTP/HTTPS requests honoring the standard `HTTP_PROXY`/`HTTPS_PROXY` environment variables. |
| **SOCKS5 Proxy via `LD_PRELOAD`** | Dynamic library calls (e.g., SMTP, Redis, Postgres clients) | `proxychains4` intercepts standard libc `connect()` calls and routes traffic to a SOCKS5 proxy. |
| **Landlock TCP OR Namespace Bridge** | Static Go/Rust binaries executing raw syscalls | A kernel-level barrier guarantees that even static binaries cannot bypass the proxies. |

### Network Backends (Linux)

Bento auto-detects kernel features to determine which isolation mode to use, falling back gracefully where necessary:

#### 1. Landlock TCP Mode (Kernel $\ge$ 6.7)
* Bento inspects the host's Landlock ABI version via `landlock_create_ruleset`. If it is $\ge$ 4, it leverages native TCP filtering.
* The script runs with `--share-net` (access to host network namespace).
* Bento spins up the local HTTP and SOCKS5 proxies on ephemeral host ports and passes their ports to the `bento-launcher` via `BENTO_ALLOW_PORTS`.
* The launcher installs Landlock rules restricting raw TCP connections (`LANDLOCK_ACCESS_NET_CONNECT_IP`) to *only* those two ephemeral proxy ports.
* Static binaries trying to connect directly to the outside world (e.g. `connect("api.example.com", 443)`) receive `EACCES` from the kernel.

#### 2. Namespace Bridge Mode (Fallback)
* Used on older kernels where Landlock TCP filtering is unavailable.
* The script is executed inside an isolated network namespace (`--unshare-net`) with no default route to the outside.
* On the host, Bento listens on Unix domain sockets connected to the proxies:
  `socat UNIX-LISTEN:/tmp/bento-http.sock TCP:localhost:<httpPort>`
* Inside the sandbox, Bento listens on fixed ports bridged back to the host sockets:
  `socat TCP-LISTEN:3128 UNIX-CONNECT:/tmp/bento-http.sock`
* Environment variables point sandbox processes to the local socket bridge.
* Static binaries trying to connect to external IPs receive `ENETUNREACH` since no network routes exist.

### Hostname Validation & Security
Bento proxies strictly validate targets to prevent DNS and address bypass techniques:
* **Canonical IPs:** Rejects shorthand literals like `127.1` or `2852039166` (which translates to `169.254.169.254`, the AWS IMDS endpoint).
* **RFC-Compliant DNS:** Prevents naive DNS matching exploits by validating that hostnames conform to RFC 1123.
* **Control Characters:** Blocks null-bytes (`\x00`) and carriage returns to defeat naive string matches.
* **IPv6 Zone IDs:** Strips and validates IPv6 format payloads to prevent bypasses.

## Execution Control (Seccomp)

By default, Bento blocks scripts from launching subprocesses (zero-config mode).

1. The `bento-launcher` shim opens the target interpreter binary as a file descriptor.
2. It sets `prctl(PR_SET_NO_NEW_PRIVS)`.
3. It registers a custom seccomp filter: denies `execve(2)` with `EPERM` while allowing `execveat(2)`.
4. It calls `execveat(fd, "", argv, envp, AT_EMPTY_PATH)` to swap the launcher process for the interpreter.

Because the interpreter is loaded via `execveat`, the initial launch succeeds. However, if the script subsequently executes a subprocess (which uses the standard `execve` syscall), the seccomp filter blocks the call, returning a `Permission Denied` error to the runtime.

### Seccomp Limitation: Inherited File Descriptors
Seccomp intercepts *system calls*, not descriptors that have already been allocated. If you allow subprocess execution (`allow_exec: true`), child processes inherit all open file descriptors from the parent. Seccomp cannot revoke access to already-open sockets or files. Developers should ensure they apply `O_CLOEXEC` to sensitive file descriptors or close them explicitly before spawning processes.

## Mandatory Deny List

To protect developers from accidentally exposing sensitive files, Bento automatically overlays hard block rules on top of any user configuration. These rules are defined in `internal/spec/dangerous.go`.

### 1. Read-Protected Paths (`spec.DangerousFiles`)
Protects credentials and private keys from exfiltration, even if the manifest declares broad read rights (e.g. `read: ["/home/user"]`):
* `~/.ssh/id_{rsa,ed25519,ecdsa,dsa,identity}`
* `~/.aws/credentials` and `~/.aws/config`
* `~/.config/gcloud/` credentials and tokens
* `~/.azure/accessTokens.json`
* `~/.kube/config`
* `~/.docker/config.json`
* `~/.git-credentials`, `~/.netrc`, `~/.npmrc`, `~/.pypirc`, `~/.gem/credentials`
* `~/.password-store`

### 2. Write-Protected Paths (`spec.DangerousWriteFiles`)
Prevents scripts from gaining persistence or altering execution environments:
* Shell profiles: `~/.bashrc`, `~/.bash_profile`, `~/.zshrc`, `~/.zprofile`, `~/.profile`
* Global configurations: `~/.gitconfig`, `~/.gitmodules`, `~/.mcp.json`, `~/.ripgreprc`

### 3. Workspace-Relative Protections
When a user grants write access to a workspace directory, Bento automatically protects repository meta-directories inside it:
* `.git/hooks/` (re-bound as read-only to prevent hook injections)
* `.git/config`
* `.vscode/tasks.json` and `.vscode/launch.json`
* `.idea/workspace.xml`

### Existing-Files-Only Limitation
Bento applies path shielding by bind-mounting `/dev/null` over target paths. Because bubblewrap requires a target file to exist on the host to mount over it, **uncreated files are not protected**. If `~/.zshrc` does not exist when Bento starts, a script with write permissions in `$HOME` could potentially create and write to it. To avoid this, always grant narrow write permissions rather than exposing whole home directories.
